# /vault-metadata-migrator — 既存ノートへのメタデータ遡及付与

新規ノートは `templates/` とスキャン系スキルが新スキーマ（CLAUDE.md「セカンドブレイン拡張フィールド」）を埋めるが、過去の蓄積は空のままである。
このスキルは既存ノートへ `summary` / `related` / `depth` を少しずつ付与し、AIが本文を開かずにVaultを読める状態へ移行する。
**まず `CLAUDE.md` を読み、絶対ルールと拡張フィールドの定義に従うこと。**

frontmatterの一括機械編集はYAML破損の主要リスクなので、次の3つを必ず守る。

1. **バッチ制**: 1回につき10〜20本。全件一括は禁止。
2. **承認制**: 適用前に「ノート / summary案 / depth / related案 / type変更の有無」の一覧を提示し、人間の承認を得る。
3. **挿入のみ**: YAMLの再シリアライズ（読み込み→dump）はしない。既存frontmatterのテキストは一切変更せず、閉じ `---` の直前に新フィールド行を**テキスト挿入**する。

## 1. バッチ計画（初期スコープ: knowledge 65本）

| バッチ | 対象 | 本数 | depth の基本値 |
| --- | --- | --- | --- |
| 1 | `Inbox/knowledge/payment/`（カリキュラム） | 14 | systematic |
| 2 | AI/エージェント技術系（[[AI-Agent-MOC]] のクラスタ1〜4） | 19 | overview |
| 3 | AI論考・キャリア系 + プロダクト系 | 16 | overview |
| 4 | 心理・行動系 + 生活・単発系 + メタノート（README、[[Concepts-MOC]]） | 残り | overview（速報由来は flash） |

`summary` が既にあるノートはスキップする（冪等）。
バッチ対象の分類は [[00_diagnosis]] と [[AI-Agent-MOC]] のクラスタを使う。

## 2. 提案の作り方

- **summary**: 本文の「一文定義」「概要」「冒頭段落」から1行に要約する。本文にない主張を足さない。`: ` を含みやすいので必ず `"..."` で囲む。
- **depth**: `flash`（速報由来）/ `overview`（単発理解）/ `systematic`（合格基準つき体系学習）。迷ったら overview。
- **related**: 実在ノートへの `"[[リンク]]"` を2〜4本。ハブ優先（所属MOC、正本ノート）+ 本文の「🔗 関連」節にある強い関連。無理に作らない。
- **as_of**: 時間で変わる情報を含むのに無い場合だけ提案する（既にある場合は触らない）。
- **type統一**: `type: note` → `type: knowledge` は**値の変更**なので、フィールド追加とは分けて承認を取る。個人的な雑記・思考メモは knowledge にしない。

## 3. 適用の手順（承認後）

Pythonスクリプトで1ファイルずつ処理する。

1. 適用前の frontmatter を `yaml.safe_load` でパースして退避（d0）。
2. 閉じ `---` の直前に新フィールド行を挿入（type変更が承認された場合のみ `type:` 行を置換）。本文には触らない。
3. 適用後を再パース（d1）し、次を検証する。失敗したら `git restore` でそのファイルだけ戻し、報告に含める。
   - d1 がパース可能
   - 既存キーの値が不変（d0 の全キーで d1 と一致。type変更承認時の type のみ例外）
   - 新キー（summary / depth / related）が存在し、related の各要素が `[[...]]` 形式
4. バッチ全体で `git diff --numstat` を取り、挿入行数が期待どおりで、削除行が type 変更分（承認時のみ）に限られることを確認する。
5. related に書いたリンク先が実在することを確認する（ファイル名ベース）。

## 4. 報告

適用本数、スキップ本数、検証結果（YAML全通過か）、git diffのサマリ、次バッチの予告を短く返す。
コミットとpushは指示があった場合のみ行う（通常はobsidian-gitに委ねる）。

## ガード

- `Claude-note/` は対象外（読みもしない）。
- リネーム・削除・本文変更はしない。frontmatterへの**行挿入**と、承認された `type:` 行の置換のみ。
- 1バッチでも検証に失敗したファイルがあれば、次バッチへ進む前に原因を報告する。
- 実行環境に PyYAML が無い場合は、構造チェック（`---` 対応、インデント、`: ` の引用）に切り替え、その旨を報告する。
