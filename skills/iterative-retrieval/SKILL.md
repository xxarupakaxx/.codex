# Iterative Retrieval — 段階的コンテキスト精製

## 概要

1回の検索で必要な情報が全て見つかることは稀。
初回検索→結果評価→不足部分の追加検索を繰り返し、漸進的に必要なコンテキストを構築する。

## トリガー

- `learnings-researcher`の結果が不十分な場合
- 複雑な調査タスクで情報が分散している場合
- サブエージェントに渡すコンテキストが不足している場合
- 自動: `exploring-codebase`や調査Phase内で暗黙的に適用

## リファインメントループ

```
Round 1: 広い検索
  ↓
評価: 「何がわかったか」「何がまだ不明か」
  ↓
Round 2: 不明点に焦点を当てた検索
  ↓
評価: 十分か？
  ↓ NO
Round 3: さらに深掘り（別の検索手法に切替）
  ↓
最大3ラウンドで終了
```

## 各ラウンドの戦略

### Round 1: 広い網

- **Grep**: 関連キーワードで広範囲検索
- **Glob**: ファイルパターンで構造把握
- **memories/solutions/**: 過去知見の横断検索

**評価テンプレート**:
```markdown
### Round 1 結果評価
- 見つかったもの: [リスト]
- まだ不明なもの: [リスト]
- 次に検索すべきもの: [具体的なキーワード/パス]
```

### Round 2: 焦点の絞り込み

Round 1で特定されたファイル・モジュールを**深く読む**:
- 関連ファイルのRead（関数定義、型定義、テスト）
- 依存先の確認（import/require先）
- git log/blameで変更履歴を確認

### Round 3: 別アプローチ

Round 2でも不足の場合:
- **外部検索**: WebSearch, deepwiki, Context7
- **人に聞く**: AskUserQuestionでドメイン知識を確認
- **推測 + 検証**: 仮説を立てて、コードで検証

## サブエージェントへの適用

`explorer`サブエージェントに以下のパターンを指示:

```
タスク: [調査内容]
検索戦略:
1. まず [キーワードA] で広く検索
2. 見つかった関連ファイルを深掘り
3. 不足があれば [代替キーワードB] で追加検索
4. 各ラウンドの結果と残りの不明点を報告
```

## `learnings-researcher`との統合

learnings-researcherが「関連度の高い結果が少ない」場合:
1. タグ・カテゴリで再検索
2. root_causeやcomponentフィールドで追加検索
3. 類似の技術スタック（同じフレームワーク等）で横断検索
