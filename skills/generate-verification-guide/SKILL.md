# ローカル検証ガイド生成スキル

## 概要

実装完了後（Phase 5）に、変更の影響範囲を探索し、ローカル環境（`pnpm dev`）での具体的な動作確認手順・テストケース・チェックリストを自動生成する。

10年後の新人エンジニアが読んでも分かるレベルの具体性を目指す。

## 実行タイミング

- Phase 5（完了報告）の一部として自動実行
- UI変更・API変更・DB変更を含む場合は**必須**
- ユーザーが明示的に `/generate-verification-guide` で呼び出す場合

## スキップ条件

- 設定ファイルのみの変更
- テストコードのみの変更
- ドキュメントのみの変更
- ライブラリアップデートのみ

## 実行手順

### Step 1: 変更ファイル一覧の取得

```bash
git diff <BASE_BRANCH>...HEAD --stat
```

変更ファイルを以下にカテゴリ分け:
- **UI（ページ/コンポーネント）**: `.tsx` in `app/` (excluding `api/`, `lib/`)
- **API**: `app/api/` 配下の `route.ts`
- **DB**: `migrations/`, `contexts/`, `infrastructure/`
- **ロジック**: `lib/`, `usecase/`, `action.ts`

### Step 2: 影響ページの探索（`explorer` サブエージェント）

変更ファイルから影響を受けるページを特定するため、`explorer` サブエージェントを起動。

プロンプト:
```
以下の変更ファイルが影響するページ（URL）を全て特定してください。
各ページについて以下を調べてください:
1. URL パス
2. 画面構成（何が表示されるか）
3. ユーザーの主要操作（クリック、入力、送信等）
4. そのページが依存するDBテーブルとAPI
5. 変更前後で動作が変わる箇所

変更ファイル:
<ファイル一覧>
```

### Step 3: テストケース生成

各影響ページについて以下のテストケースを生成:

#### 必須テストケース
- **正常系**: データがある状態での基本操作
- **空データ系**: テーブルが空の場合の表示
- **エラー系**: API キー未設定・認証切れ等でのエラー表示

#### UI変更がある場合の追加テストケース
- 新しいボタン/フォーム/セクションの表示確認
- インタラクション（クリック→確認→結果表示）
- レスポンシブ表示（必要に応じて）

#### DB変更がある場合の追加テストケース
- マイグレーション実行の確認コマンド
- テーブル/カラムの存在確認コマンド
- データの投入・確認コマンド

### Step 4: チェックリスト生成

全テストケースを `- [ ]` 形式のチェックリストにまとめる。

### Step 5: 出力

メモリディレクトリの `90_verification.md` に以下の構造で保存:

```markdown
# ローカル検証ガイド

## 前提条件
- 必要な環境変数
- 必要な外部サービス
- データの前提

## 影響ページ一覧
| ページ | URL | 変更内容 | 確認優先度 |
|--------|-----|---------|-----------|

## 検証手順

### 1. 環境セットアップ
（マイグレーション、データ投入等）

### 2. ページ別テストケース

#### [ページ名]（URL）
**画面構成**: ...
**テストケースA: 正常系**
1. 操作手順（具体的な入力値を含む）
2. 期待結果（画面にどう表示されるか）
**テストケースB: 空データ**
...

### 3. DB確認コマンド
（レコード数確認、状態確認等）

## チェックリスト
- [ ] ...
- [ ] ...

## トラブルシューティング
### Q: ...
```

### Step 6: HTMLビューア表示（オプション）

検証ガイドをインタラクティブなHTMLビューアで表示する。チェックリストの進捗トラッキング付き。

```
mcp__workflow-html-app__view-verification(
  content: "<90_verification.mdの内容>"
)
```

- ユーザーが明示的にHTMLビューアを要求した場合に使用
- チェックボックスをクリックすると進捗バーが更新される
- 検証項目の完了状況を視覚的に確認できる

## 出力の品質基準

- **具体性**: 「検索する」ではなく「入力欄に『データの暗号化』と入力して送信ボタンをクリック」
- **期待結果の明示**: 「動作する」ではなく「緑色のChipに『同期を開始しました』と表示される」
- **前提条件の明示**: どのテーブルにデータが必要か、embedding が必要か
- **エラーケースの網羅**: API キー未設定時、データなし時の挙動
- **DB確認コマンド**: docker exec で実行できるSQL
- **トラブルシューティング**: よくあるエラーと対処法
