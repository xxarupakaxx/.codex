# QA セッション

対話型の QA セッションを実行します。
ユーザーが遭遇している問題を説明し、こちらは必要最小限の確認を行い、文脈を得るためにコードベースを探索し、長持ちしてユーザー視点で読める GitHub Issue を起票します。

## ユーザーが挙げた各 issue について

### 1. 聞き取りと軽い確認

まずはユーザー自身の言葉で問題を説明してもらいます。
確認質問は **短く 2〜3 個まで** にとどめ、次の点だけに絞ります。

- 期待していたことと、実際に起きたことは何か。
- 再現手順は何か。
- 常に起きるのか、断続的なのか。

聞き取りをやり過ぎてはいけません。
起票できるだけの情報がそろっているなら、そのまま先へ進みます。

### 2. 背景でコードベースを探索する

ユーザーと話しながら、関連箇所を理解するために背景で探索 sub-agent を起動します。
Claude Code では Agent（`subagent_type=Explore`）を使います。
Codex では `multi_agent_v1.spawn_agent` または現在利用できる agent orchestration で explorer 相当の role を使います。
目的は修正案を見つけることではなく、次の 3 点です。

- その領域で使われているドメイン言語を把握すること（`UBIQUITOUS_LANGUAGE.md` を確認する）。
- その機能が本来何をするはずなのかを理解すること。
- ユーザーに見える振る舞いの境界を特定すること。

この文脈は issue をより良く書くために使います。
ただし issue 本文には、具体的なファイル名、行番号、内部実装の詳細を書いてはいけません。

### 3. スコープ判定をする

起票前に、これは **単一 issue** なのか、それとも **複数 issue に分解** すべきかを判断します。

次のときは分解します。

- 修正が複数の独立した領域にまたがるとき。
- 別の人が並行して進められる、明確に分離可能な論点があるとき。
- ユーザーの説明に、明確に別個の故障モードや症状が複数含まれているとき。

次のときは単一 issue のままにします。

- 一か所の一つの振る舞いが誤っているだけのとき。
- 症状がすべて同じ根本挙動に由来しているとき。

### 4. GitHub Issue を起票する

`gh issue create` で issue を作成します。
起票前レビューをユーザーに求めず、そのまま作って URL を共有します。

Issue は **長持ちする内容** でなければいけません。
大きなリファクタ後でも意味が通るように、ユーザー視点で書きます。

#### 単一 issue の場合

次のテンプレートを使います。

```md
## 起きたこと

[ユーザーが実際に体験した振る舞いを、平易な言葉で書く]

## 期待していたこと

[期待していた振る舞いを書く]

## 再現手順

1. [開発者が追える具体的な手順を書く]
2. [内部モジュール名ではなく、コードベースのドメイン用語を使う]
3. [関連する入力値、フラグ、設定があれば含める]

## 補足

[ユーザーの観察やコードベース探索から得た補足を書く。例として「filesystem layer ではなく Docker layer を使うときだけ起きる」のように、ドメイン言語で書き、ファイルは引用しない]
```

#### 分解する場合

依存順に issue を作成します。
まず blocker を先に作ることで、後続 issue から実際の issue 番号を参照できます。

各 sub-issue には次のテンプレートを使います。

```md
## 親 issue

#<parent-issue-number>（追跡用 issue を作った場合）または「QA セッション中に報告」

## 問題の内容

[この slice だけに絞って、具体的な振る舞いの問題を書く]

## 期待していたこと

[この slice に対する期待挙動を書く]

## 再現手順

1. [この issue 専用の手順を書く]

## ブロック要因

- #<issue-number>（この issue が別 issue の解決なしには直せない場合）

依存がなければ「None — can start immediately」と書きます。

## 補足

[この slice に関係する補足を書く]
```

分解して起票するときのルールは次のとおりです。

- **太い issue を少なく作るより、薄い issue を多く作るほうを優先します。**
- **依存関係は正直に書きます。**
- **依存順に issue を作成します。**
- **並列に着手できる状態を最大化します。**

#### すべての issue 本文に共通するルール

- **ファイルパスや行番号は書きません。**
- **プロジェクトのドメイン言語を使います。**
- **コードではなく振る舞いを記述します。**
- **再現手順は必須です。**
- **簡潔に保ちます。**

起票が終わったら、すべての issue URL を表示し、依存関係を要約して共有します。
そのあとで「次の issue に進みますか、それともここで終えますか」と聞きます。

### 5. セッションを継続する

ユーザーが終わりと言うまで続けます。
各 issue は独立して扱い、まとめて処理しません。
