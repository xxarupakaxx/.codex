# 優れたスキルを作るための用語集

優れたスキルを構成する要素のドメインモデルである。
スキルは確率的なシステムから決定性を引き出すために存在し、その根本的な価値は**予測可能性**にある。
以下の用語はすべて、それを調整する手段である。
これは [`writing-great-skills`](SKILL.md) が段階的に開示する参照資料である。

用語は、**起動**（スキルへ到達する方法）、**情報階層**（内容の配置方法）、**誘導**（実行時のエージェントの振る舞いを形作る方法）、**剪定**（簡潔さを保つ方法）という軸で分類する。
各**失敗モード**は、それを解消する手段の近くに置き、_失敗モード_ と記す。

定義内の**太字の用語**もこの用語集で定義している。
該当する見出しを探す。

## 予測可能性

スキルが毎回、エージェントに同じ**やり方**で振る舞わせる度合いである。
同じ出力ではなく、同じプロセスを指す。
ブレインストーミングのスキルは予測可能な形で発散すべきであり、生成するトークンは変わっても振る舞いは変わらない。
ほかのすべての用語が支える根本的な価値である。
コストと保守性は競合する価値ではなく、その表れである。

_避ける語_：一貫性、信頼性、堅牢性、出力の決定性

## 起動

スキルへ到達する方法と、その選択によって支払う二種類の負荷を指す。

### モデル起動

**description** フィールドを保持し、エージェントが認識して自律的に起動できるスキルである。
人間も名前を入力できるため、モデル起動には常にユーザーからの到達も含まれる。
モデル専用の状態は存在しない。
description はエージェントによる発見を追加するだけで、人間からの到達を除かない。
発見可能性と引き換えに、毎ターン恒久的な**コンテキスト負荷**を支払う。
エージェントが発見できる description があるため、ほかのスキルからも起動できる。
内容がすべて**参照情報**のモデル起動スキルは、共有参照情報の置き場所にもなる。
複数のスキルが必要とする情報を一か所に置き、ほかのスキルから起動できるためである。
エージェントが自力で到達する必要がある場合だけモデル起動を選ぶ。
手動でしか起動しないなら description を外し、コンテキスト負荷をなくす。

_避ける語_：能力、ツール、ケイパビリティ

### ユーザー起動

**description** を外し、エージェントから見えず、人間が名前を入力した場合だけ到達できるスキルである。
**モデル起動**がユーザーとエージェントの両方から到達できるのに対し、ユーザーだけが起動できる。
エージェントによる発見可能性と引き換えに、**コンテキスト負荷**をゼロにする。
description がないため、人間以外は到達できず、ほかのスキルからも起動できない。

_避ける語_：手順、ワークフロー、コマンド

### Description

スキルを起動する機械可読な条件であり、**モデル起動**スキルが常に読み込むことを強制される唯一の**コンテキストポインター**である。
description の有無が起動方式を決める。
保持すればモデル起動になり、ほかのスキルからも到達できる。
削除すれば**ユーザー起動**になり、人間だけが到達できる。
モデル起動スキルの**コンテキスト負荷**の源である。

_避ける語_：frontmatter、要約

### コンテキストポインター

コンテキスト外の資料を指し、それへ到達する条件を記述した、エージェントのコンテキスト内にある参照である。
**description** は最上位のコンテキストポインター（コンテキストウィンドウからスキル）であり、開示されたファイルへのポインターは一段下にある同種のものである。
対象ではなくポインターの文言が、エージェントが**いつ**、どれほど**確実に**到達するかを決める。
必須資料を弱い文言のポインターの後ろに置くと、振る舞いのばらつきが生じる。
最初に文言を強め、それでも失敗する場合だけ資料をインラインへ戻す。

_避ける語_：リンク、参照、import

### コンテキスト負荷

**モデル起動**スキルがエージェントのコンテキストウィンドウへ課すコストである。
常に読み込まれる **description** がトークンと注意を消費する。
**ユーザー起動**スキルは description を持たないため、この負荷を免れる。
モデル起動スキルを細かく分割しすぎないための制約にもなる。

_避ける語_：トークンコスト、コンテキストの肥大化

### 認知負荷

**ユーザー起動**スキルが人間へ課すコストである。
存在するスキルと、それぞれを使う時機を人間が覚え、人間自身が索引になる。
**モデル起動**は、エージェントが発見できるため、この負荷を取り除く。
ユーザー起動スキルを細かく分割しすぎないための制約にもなる。
ただし、最小化すべきコストではない。
これは人間の主体性の代価であり、一部のスキルをユーザー起動に保つ理由である。
人間の判断が重要な場所では支払い、重要でない場所では取り除く。

_避ける語_：人間の索引、負担、オーバーヘッド

### ルータースキル

ほかのユーザー起動スキルの名前と使う時機を示し、人間が多数ではなく一つだけ覚えればよいようにする**ユーザー起動**スキルである。
提案はできるが、ほかのスキルを起動することはできない。
ユーザー起動スキルには **description** がなく、人間以外は到達できないためである。
ユーザー起動スキルが増えたときの**認知負荷**を解消する。

_避ける語_：dispatcher、menu、registry、index、router procedure

### 粒度

スキルをどれほど細かく分けるかを表す。
細分化すると、二種類の負荷のどちらかが増える。
**モデル起動**スキルを増やすと、description がウィンドウを占有し、注意を奪い合うため、**コンテキスト負荷**が増える。
**ユーザー起動**スキルを増やすと、人間が覚えて呼び出す対象が増えるため、**認知負荷**が増える。
分割には二種類ある。
**起動**による分割では、プロンプトで実際に使う固有の**先導語**があり、それを起動条件にできる場合にモデル起動スキルを分ける。
**順序**による分割では、現在のステップに続く**完了後のステップ**を隠す必要がある場合に、一連の**ステップ**を分ける。
独立したコンテキストへ隔離すると、後続の内容を消せるためである。
逆方向にも注意する。
手順を統合すると、各ステップから後続のステップが見えるようになり、早すぎる完了を招く。

_避ける語_：chunking、modularity

## 情報階層

スキルの内容をどのように配置し、各情報を階層のどこへ置くかを表す。

### 情報階層

エージェントが情報を必要とする即時性でスキルの内容を順位付けした一つの階層である。
ファイル内かポインターの後ろか、ステップか参照情報かという二つの切り分けから、次の段が生まれる。

- **ステップ**：ファイル内の主要層
- **参照情報**：ファイル内の副次層
- **参照情報**：**コンテキストポインター**の後ろで開示される層

**ステップ**がないスキルは下の二段だけを使う。
レビューのすべてのルールを同じ段へ置く場合のように、妥当な同格集合になることも多く、問題ではない。
情報階層は起動方式から独立している。
全体がステップ、全体が参照情報、その混合のいずれでも、モデル起動にもユーザー起動にもできる。
ステップがあるスキルで、開示すべきファイル内参照情報が残っていると、ステップが埋もれ、注意が向くかどうかが偶然に左右される。
これは可読性だけでなく、振る舞いのばらつきを左右する。
階層の上部を読みやすく保ち、下げられる情報は下へ移す。

_避ける語_：structure、organization、layout

### ステップ

エージェントが実行する順序付きの操作である。
スキルにステップがある場合は内容の主要層となり、SKILL.md に置く価値がある部分になる。
すべてのスキルにステップがあるわけではない。
起動方式にかかわらず、すべてがステップ（`tdd`）、すべてが**参照情報**（レビュー）、または両方の混合になりうる。
すべてのステップは、明確か曖昧かを問わず、**完了基準**で終わる。

_避ける語_：workflow、instructions、choreography

### 参照情報

エージェントが必要に応じて参照する、定義、事実、パラメーター、例、条件付き指示である。
スキルに**ステップ**がある場合は副次層、なければ内容全体になる。
スキルの外部に置くこともできる。
**外部参照情報**を参照する。
**コンテキストポインター**を通じて到達し、**段階的開示**の主要な候補になる。

_避ける語_：補足資料、docs、背景

### 外部参照情報

スキルシステムの外にある**参照情報**である。
description も**ステップ**もなく、起動できない通常のファイルであり、どのスキルからも指せる。
自力で起動する必要がない共有参照情報の置き場所になる。
また、二つの**ユーザー起動**スキルが共有できる唯一の置き場所でもある。
どちらにも description がなく、互いを起動できないためである。

_避ける語_：doc、resource、knowledge base

### 段階的開示

**参照情報**を SKILL.md の外へ出し、**コンテキストポインター**の後ろへ移して、階層を下げることである。
階層上部の読みやすさを保つ。
主目的はトークンの最適化ではなく、**情報階層**の保護にある。
**分岐**が判断基準になる。
一部の分岐だけが必要とする情報は開示し、すべての経路が必要とする情報はインラインに置く。
必須情報へのポインターが安定して発火しない場合は文言を強め、それでも失敗する場合だけインラインへ戻す。

_避ける語_：lazy loading、chunking

### 併置

エージェントが同時に必要とする資料を一か所にまとめることである。
概念の定義、ルール、注意点を一つの見出しの下に置き、ファイル全体へ散らさない。
一部を読めば、その近くにある関連情報も読めるようになる。
**情報階層**をファイル内で補完する。
情報階層が情報を**どこまで下げるか**を決めるのに対し、併置は下げた場所で**何を隣に置くか**を決める。
一連の**参照情報**に適した形式を導く公式はない。
スキルがエージェント向けの文書として読めるかを判断基準にする。
散在した資料より、まとまった資料の方が文書として読みやすい。
**重複**とは異なる。
重複は一つの意味を二か所で繰り返すが、散在は一つの意味を多数の場所へ分断する。

_避ける語_：grouping、clustering、cohesion

### 肥大化

_失敗モード。_
古いか重複しているかに関係なく、SKILL.md の行数が単純に多すぎる状態である。
すべてが現役で固有の内容でも肥大化しうる。
エージェントが動く前に読む量が増え、余分な内容へ注意が薄まるため、可読性を損なう。
各行を**関連性**のある状態に保つ必要があるため、保守性も損ない、トークンも消費する。
**情報階層**で解消する。
**参照情報**を**コンテキストポインター**の後ろへ下げ、**分岐**または順序で分け、各経路が必要なものだけを持つようにする。
古い内容の蓄積による長さである**堆積**や、同じ意味の繰り返しによる長さである**重複**とは異なる。
肥大化は、原因を問わず長さそのものである。

_避ける語_：bloat、length、size、verbosity

## 誘導

エージェントの実行時の振る舞いを**予測可能性**へ向けて形作る手段である。

### 分岐

スキルを起動する方法のうち、ほかと区別されるものを指す。
スキルが扱うケースごとに、実行時の経路が変わる。
ステップが多いスキルは多数の分岐を持つ場合があり、直線的なスキルには分岐がない。

_避ける語_：path、case、fork

### 先導語

モデルの事前学習にすでに存在し、スキルの実行中にエージェントが考えるために使う、_Leitwort_ とも呼ばれる簡潔な概念である。
モデルがすでに持つ事前知識を呼び起こし、最小限のトークンで振る舞いの原則を表現する（例：_lesson_、_proximal zone of development_、_fog of war_、_tracer bullets_）。
文ではなくトークンとして繰り返すと、スキル全体に分散した定義が積み上がり、振る舞いの広い領域を固定する。
明確に定義すれば新しい語を作ることもできるが、造語は事前知識を呼び起こさない。
事前学習済みの語なら無料で得られる効果を、定義用のトークンで支払うことになる。
まず既存の語を探す。

先導語は**予測可能性**へ二つの効果を与える。
本文では**実行**を固定する。
概念が現れるたびに同じ振る舞いを選び、平坦な参照情報では調べる対象の種類へ注意を向け、毎回適切な検査を引き出す。
**description** では**起動**を固定する。
同じ語がプロンプト、文書、コードベースにも存在すると、エージェントはその共有語彙をスキルへ結び付け、より安定して起動する。
スキルを使いたいときに実際に使う先導語を description に含める。

_避ける語_：keyword、term、motif

### 完了基準

作業単位が完了したとエージェントへ伝える条件であり、判断対象となる目標である。
二つの性質によって、単なる品質ではなく制御手段になる。
第一は**明確さ**である。
完了と未完了を判別できれば、**早すぎる完了**に抵抗できる。
「理解に達した」のような曖昧な境界は、エージェントが完了を宣言して次のステップへ移る余地を与える。
早すぎる完了はステップ間の失敗であるため、この性質が働くには**ステップ**が必要になる。
第二は**要求水準**である。
必要な**下準備**の量を決める。
「変更したモデルをすべて説明する」は徹底的な作業を強制するが、「変更一覧を作る」は強制しない。
要求水準はステップに限定されず、平坦な参照情報にも適用できる。
そのため、ステップのないスキルでも「すべてのルールを適用する」という網羅性の基準を持てる。
最も強い基準は、検査可能で網羅的である。

_避ける語_：done condition、exit condition、stopping rule

### 下準備

一つのステップの中でエージェントが裏側で行う作業である。
ファイルを読み、コードベースを調べ、変更を加え、必要なものをユーザーへ押し付けず自分で探す。
ステップ構造より下にあり、独立したステップとしては書かれない。
文言に潜在し、スキルではなくエージェントが制御する。
ステップをまたいで作用する**完了後のステップ**の引力に対し、下準備はステップ内の対応物である。
_comprehensive_ や _thorough_ のような**先導語**、または網羅的な作業を求める**完了基準**によって増える。
要求水準は平坦な参照情報にも適用でき、すべての段を扱うよう駆動する。
要求がない場合や、**早すぎる完了**によってステップが短縮された場合は薄くなる。

_避ける語_：scope、effort、diligence、coverage

### 完了後のステップ

現在のステップに続く**ステップ**である。
見えていると、エージェントを前へ引っ張り、**早すぎる完了**を招く。
見える数が多いほど引力が強くなる。
一連のステップを二つに分けて隠すことで防ぐ。

_避ける語_：horizon、fog of war、lookahead

### 早すぎる完了

_失敗モード。_
作業そのものではなく完了することへ注意が移り、現在のステップが本当に終わる前に終了する状態である。
ステップ間の失敗なので、発生には**ステップ**が必要になる。
ステップのないスキルが早く終了する場合は早すぎる完了ではなく、要求を満たさない薄い**下準備**である。
見えている**完了後のステップ**が前へ引く力と、**完了基準**の明確さが抵抗する力の綱引きで起きる。
明確で検査可能な基準は耐えるが、曖昧な基準は崩れる。
曖昧さが必要条件である。
境界が明確なら、後のステップがいくつ見えても引力に耐えられるため、急がないステップを防御する必要はない。
二つの手段を順に使う。
最初に**境界を明確にする**。
局所的でコストが低い。
基準から曖昧さを除けず、実際に急ぐ挙動を観測した場合だけ、**後のステップを隠す**。
隠す方法は、ユーザー起動の引き継ぎや subagent への委任など、実際のコンテキスト境界をまたぐ場合だけ機能する。
インラインのモデル起動呼び出しでは後のステップがコンテキストに残り、何も消えない。
薄い下準備の原因の一つだが、同じものではない。
ステップを最後まで実行しても下準備が薄い場合はある。

_避ける語_：premature closure、the rush、rushing、shortcutting

### 否定

_失敗モード。_
エージェントへ「しないこと」を伝える禁止による誘導である。
禁止した振る舞いをコンテキストへ持ち込み、利用可能性を下げるのではなく上げてしまう。
「象を考えるな」と言えば象だけが残り、「冗長なコメントを絶対に書くな」と言えば、エージェントは冗長さというパターンを読んだ直後になる。
否定は弱い修飾であり、強く活性化した概念に負けるため、禁止が対象を実行する指示のように半ば読まれる。
その**先導語**は、禁止が枠内へ持ち込む「象」である。
**肯定形**で促すことで解消する。
「一行のコメントを書く」のように目標の振る舞いを示し、禁止対象へ触れない。
肯定形にできない振る舞いへの強いガードレールとしてだけ禁止を置く。
その場合も肯定的な目標を併記し、行うべきことへ注意を向ける。

_避ける語_：ironic rebound、don't-prompting、the pink elephant

## 剪定

スキルを簡潔に保つ方法であり、各対策を解消対象の失敗へ対応させる。

### 信頼できる唯一の情報源

各意味が権威ある一か所だけに存在し、スキルの振る舞いを変えるときに一か所だけ編集すればよい状態である。
**重複**はこの状態に違反する。

_避ける語_：home、canonical location

### 重複

_失敗モード。_
同じ意味に複数の**信頼できる唯一の情報源**を与える状態である。
一か所を変更したらほかも変える必要があるため保守コストが増え、トークンも消費する。
意味を繰り返すと、情報階層における本来の順位以上に目立つようになる。
意味ではなくトークンを意図的に繰り返して注意を高める**先導語**とは、意図せず逆の関係になる。

_避ける語_：repetition、redundancy

### 関連性

ある行が、スキルの動作に今も関係しているかを表し、残す内容を判断する基準になる。
単なる解説や開示すべき**分岐**のように、最初からタスクに関係しない場合は関連性を失う。
記述する振る舞いや世界が変わり、内容が古くなった場合にも失う。
短いスキルは各行を確認するコストが低いため、関連性を保ちやすい。
**無作用**とは異なる。
関連性は行がタスクに関係するかを問い、振る舞いを変えるかは問わない。

_避ける語_：load-bearing、staleness、freshness

### 堆積

_失敗モード。_
追加は安全に感じ、削除は危険に感じるため、古い内容を取り除かずスキルへ積み重ねる状態である。
古く関連性のない行が蓄積し、現役の内容を探すには層を掘り抜く必要がある。
剪定の規律がないスキルが行き着く既定の状態である。
同じ意味を繰り返す**重複**に対し、堆積は**関連性**を徐々に侵食する。

_避ける語_：accretion、bloat、cruft、rot

### 無作用

_失敗モード。_
モデルが既定ですでに行うため、何も変えない指示である。
エージェントがどのみち行うことを伝えるために負荷を支払う。
その行は既定と比べて振る舞いを変えるか、と検査する。
完全な**関連性**があっても無作用になりうる。
**先導語**を無料にする事前知識は、無作用を無価値にもする。

先導語は**手法**であり、無作用は行への**判定**であり、両者は交差する。
既定の挙動を超えるには弱すぎる先導語は無作用になる。
エージェントがすでにある程度徹底的なら _be thorough_ は無作用であり、解決策は別の手法ではなく、判定を通る強い語である _relentless_ へ変えることになる。
したがって、既定と比べて振る舞いを変えるかという無作用の検査は、先導語が繰り返しに値するかを評価する方法にもなる。
これは読み手ではなくモデルに依存する。
ある行が無作用かについて二人が意見を異にする場合、既定の挙動について意見が異なっている。
議論ではなく、スキルを実行して決着させる。

_避ける語_：redundant instruction、restating the obvious、belaboring
