# Claude Code プロンプト設計パターン

Claude Code固有の設計パターン集。CLAUDE.md、skills、hooks、サブエージェントプロンプトの実践的なガイドライン。

## CLAUDE.md設計

### 指示数の上限

Claudeは約150-200の指示を確実に遵守できる。それ以上は競合し注意が分散する。追加する各指示は他のすべてと注意を奪い合う。

### 設計原則

1. **「何を」ではなく「なぜ」を説明**: なぜそうすべきかの文脈がClaudeの判断力を向上させ、予期しない状況でも適切に行動できる
2. **継続的更新**: `#`キーで作業中にClaudeが追加提案
3. **60行ルール**: メインファイルは60行以下。詳細は`@`参照でcontext/に委譲
4. **明確な契約**: 良いシステムプロンプトは「短い契約」のように読める—明示的、有界、検証可能

### 効果的な指示の書き方

**具体的に指示する:**

- ✅ 具体的: 「レスポンスを2-3文に制限」
- ❌ 曖昧: 「簡潔に」

**肯定形で指示する:**

- ✅ 肯定形: 「散文段落で書いて」
- ❌ 否定形: 「markdownを使うな」

**文脈を与える:**

- ✅ 文脈付き: 「TTS読み上げのため省略記号を使わない」
- ❌ 文脈なし: 「省略記号を絶対使うな」

**段階的な番号付きリストを使う:**

- 手順の順序や完全性が重要な場合、番号付きリストまたは箇条書きで指示

### 強調表現の使い分け（Claude 4.x対応）

| 表現 | 用途 | 注意 |
|------|------|------|
| 通常の文 | 一般的なガイダンス | Claude 4.xはこれで十分に従う |
| **IMPORTANT** | 重要だが例外あり | 適度に使用 |
| **CRITICAL** | 絶対に従うべき | 本当に必要な場合のみ |
| ~~YOU MUST~~ | 強制 | Claude 4.xではovertriggerリスク。通常の指示で代替推奨 |

## Skills設計

### YAML Frontmatter

```yaml
---
name: skill-name
description: トリガー条件と用途を明確に記述。
  どんな場合に使うか、どんなキーワードでトリガーされるかを含める。
---
```

### 自由度の設計

| 自由度 | 使用場面 | 例 |
|-------|---------|-----|
| 高 | 複数アプローチが有効 | コードレビュー |
| 中 | 推奨パターンあり | レポート生成 |
| 低 | 脆弱な操作、一貫性重要 | DBマイグレーション |

### スキル内のプロンプト構造

```
[ロール/コンテキスト定義]
→ [具体的なステップ/手順]
→ [出力フォーマット/テンプレート]
→ [制約/禁止事項]
→ [例（必要な場合）]
```

## Hooks設計

- **Stop hook**: レビューサブエージェントをトリガーする主パターン
- **スコープ**: スキルのライフサイクルにスコープ（スキルがアクティブなときのみ実行）
- **ファイル変更の自動検証**: 変更されたファイルに対してlint/format/typecheckを自動実行

## サブエージェントプロンプト設計

### 必須要素

1. **タスクを明確に**: 何を調査/実行するか具体的に記述
2. **対象ファイルのフルパスを渡す**: 曖昧な参照を避ける
3. **出力形式を指定**: 構造化された結果を要求
4. **コンテキストを最小化**: 必要な情報のみ渡す（ウィンドウの公共財原則）

### Overtrigger対策

Claude 4.xのsubagent orchestrationは強力だが、過剰使用に注意:

- 単純なgrep/glob/readで済む場合はsubagentを使わない
- subagentが適切な場面: 並列実行可能、独立コンテキスト必要、独立ワークストリーム
- subagentが不適切な場面: 単一ファイル編集、逐次操作、ステップ間のコンテキスト共有が必要

### プロンプト例: レビューサブエージェント

```markdown
以下のファイルをセキュリティ観点でレビューしてください。

対象ファイル:
- /path/to/file1.ts
- /path/to/file2.ts

レビュー観点:
- SQLインジェクション
- XSS
- 認証・認可の不備
- IDOR（パラメータレベルの認可チェック）

API routeが呼び出すusecase/entity定義:
- /path/to/usecase.ts
- /path/to/entity.ts

出力形式:
各指摘を以下の形式で報告:
- 重要度: critical / major / minor / nit
- ファイル: パス:行番号
- 問題: 概要
- 修正案: 具体的なコード変更
```

## エージェント設計パターン

### Orchestrator-Worker

複雑なタスクで最適なサブタスクが事前に予測できない場合:

1. Orchestratorがタスクを分析・計画
2. 動的にWorkerにサブタスクを委任
3. 結果を統合

### Reflection（自己修正）

生成 → レビュー → 修正の3ステップ:

1. ドラフト生成
2. 基準に照らしてレビュー
3. レビュー結果に基づいて修正

各ステップを別APIコールにすると、中間出力の検査・分岐が可能。

### Plan-and-Execute

高性能モデルで計画 → 軽量モデルで実行（コスト最大90%削減）:

1. Frontier model: 戦略・計画の作成
2. Mid-tier model: 標準タスクの実行
3. Small model: 高頻度の定型処理
