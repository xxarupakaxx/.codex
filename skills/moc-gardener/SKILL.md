# /moc-gardener — MOCの地図を最新に保つ

MOCは作った時点から古び始める。
このスキルは、知見ノートがテーマの入口（MOC）から辿れる状態を機械的に点検し、接続漏れを繋ぐ。
**まず `CLAUDE.md` を読み、絶対ルール（追記のみ、新規はInbox配下、Claude-note不可侵）に従うこと。**

週次の [[weekly-learning-review]] は当週の新規ノートだけを見る。
このスキルは全量を見る月次または随時の深い手入れとして使い分ける。

## 1. 到達性の判定

- MOC集合: [[AI-Agent-MOC]]、[[LayerX入社準備-MOC]]、[[Concepts-MOC]]、[[Paymentドメイン基礎マップ]]（増えたらここに追記する）。
- 対象: `Inbox/knowledge/` 配下の全ノート（READMEとMOC自身を除く）。
- 判定: 全ノートの `[[wikilink]]`（frontmatterのrelated含む）から**無向グラフ**を作り、MOC集合からの距離を測る。
  - 距離1: MOCと直接リンク。
  - 距離2: MOC直結ノートの隣（概念ノート経由のハブ&スポークは良い設計なので容認）。
  - **距離3以上または到達不能**: 接続漏れ候補。
- 判定はスクリプトで行い、目視で数えない。

## 2. 接続漏れ候補の処遇

候補ごとに次のどれかを判断する。

1. **接続する**: 内容が既存MOCのクラスタに合う場合。MOC側のクラスタへ1行追記し、ノート側の末尾「🔗 関連」にもMOCへのリンクを追記して双方向にする（herdr方式）。
2. **中継ノートに繋ぐ**: MOC直結は大げさだが、概念的に近い距離1〜2のノートがある場合。両ノートの「🔗 関連」に相互リンクを追記する。
3. **容認する**: 単発の技術メモや生活メモで、無理にリンクを作ると地図が濁る場合。容認リストとして実行記録に残す（次回以降も再提案しない）。

対話中なら1と2は承認を得て適用し、無人実行なら提案に留める。
本文と無関係なリンクを作らないこと（合格基準の「正本を1つにし他はリンクで参照」を守る）。

## 3. 実行記録

`Inbox/automation/gardener/gardener-YYYY-MM-DD.md` に、到達性の統計（距離別本数）、適用した接続、容認リストを記録する。
frontmatterは拡張スキーマ（summary、depth: overview、related に手入れしたMOC）を使う。

## 4. 報告

到達性の統計、適用件数、容認件数、記録ノートへのリンクを短く返す。

## ガード

- 既存ノートへの変更は「🔗 関連」節への行追記とMOCクラスタへの行追記のみ。
- リンク先の実在をファイル名ベースで検証してから書き込む。
- リネーム・削除・本文の書き換えはしない。

## ⏰ スケジュール設定

- **主モードは on-demand**。定期化するなら月1回（ヘルスチェックと同じタイミング）が目安。登録は人間が [[SCHEDULES]] の手順で行う。
