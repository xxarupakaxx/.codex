# Strategy Guide — Refine / Pivot 判定ガイド

## 戦略判定フローチャート

```
スコアを確認
  │
  ├─ ≥25/35 ──────────────────────────────→ PASS
  │                                          ループ終了
  │
  ├─ <25 かつ前回比 +2以上 ────────────────→ REFINE
  │     方向性は正しい。批評に従い修正      「この方向で合っている」
  │
  ├─ <25 かつ前回比 ±1以内 ────────────────→ 停滞カウント++
  │     │
  │     ├─ 停滞1回目 ─→ REFINE（もう1回試す）
  │     ├─ 停滞2回目 ─→ PIVOT（方向転換）
  │     └─ 停滞3回目 ─→ ESCALATE（ユーザーに判断を委ねる）
  │
  ├─ <25 かつ前回比 -2以上 ────────────────→ REVERT + REFINE
  │     前回の方が良い。git revertして       「前回に戻って別を試す」
  │     別の改善アプローチを試す
  │
  └─ PIVOT回数 ≥ 上限（デフォルト2） ─────→ ESCALATE
        これ以上の方向転換は非効率             「ユーザーに相談」
```

---

## REFINE（洗練）

**いつ**: スコアが上昇傾向。現在のデザイン方向性が機能している。

**Generatorへの指示テンプレート**:

```
## Iteration {N} — REFINE

前回のEvaluator批評に基づき、以下を修正してください。

### KEEP（絶対に変更しない）
{Evaluatorの KEEP リストをそのまま転記}

### FIX（必ず修正）
{Evaluatorの FIX リストをそのまま転記}

### CONSIDER（可能なら対応）
{Evaluatorの CONSIDER リストをそのまま転記}

### 注意
- KEEPリストの要素を壊さないこと
- 1つの修正が他の良い部分に悪影響を与えないか確認すること
- 修正は最小限に。過剰な変更は新たな問題を生む
```

---

## PIVOT（方向転換）

**いつ**: スコアが停滞。現在のアプローチでは限界に達した。

**PIVOTの本質**: 段階的改善ではなく、根本的に異なるアプローチを試す。
ブログ記事の例: 9回目のダークテーマLPが10回目で3Dギャラリー空間に変貌。

**Generatorへの指示テンプレート**:

```
## Iteration {N} — PIVOT

これまでのアプローチでスコアが停滞しています（{過去3回のスコア}）。
根本的に異なるデザインアプローチで再構築してください。

### 現アプローチ（捨てるもの）
{現在のデザインの特徴を3-5行で要約}

### 転換の方向性（以下から選択 or 独自提案）
1. レイアウトの根本変更: グリッド→フリーフォーム、スクロール→空間ナビゲーション
2. 美学の転換: ミニマル→マキシマリスト、フラット→3D、ダーク→ライト
3. インタラクションの革新: 従来のクリック→ドラッグ、スクロール→ジェスチャー
4. メタファーの変更: デスクトップ→物理空間、リスト→マップ、ページ→部屋

### 制約
- Phase Aで決定した製品仕様（機能要件）は維持すること
- 以前のPASS要素で再利用可能なものがあれば活用してよい
- ただし、見た目の印象は根本的に異なること
```

### PIVOTの判断基準

| 指標 | PIVOT推奨 | PIVOTしない |
|------|----------|------------|
| 2回連続±1以内 | ✅ | |
| Design or Originality が1-2で停滞 | ✅ | |
| Craft/Functionは高いがDesignが低い | ✅ 美学の問題 | |
| 全基準が均等に3前後 | ✅ 方向性の問題 | |
| Design高いがCraft低い | | ✅ 実装精度の問題（REFINE） |
| 特定1基準だけ低い | | ✅ 局所的修正で対応可能 |

## REVERT（巻き戻し）

**いつ**: 前回より明確に悪化（-2以上）。修正が裏目に出た。

**手順**:
1. `git diff HEAD~1` で前回からの変更を確認
2. `git revert HEAD` で前回の状態に戻す（またはgit stashで一時退避）
3. Evaluatorの前回のFIXリストを再確認し、別のアプローチで修正

---

## ESCALATE（ユーザーエスカレーション）

**いつ**: 自律的な改善が限界に達した。

**AskUserQuestion テンプレート**:

```
デザインループが{N}回の反復後、スコアが停滞しています。

現在のスコア推移: {scores.mdの直近5行}
最終スクリーンショット: {パス}

以下の選択肢があります:
1. 現在の状態で完了（スコア {N}/35）
2. 特定の方向性を指示して継続
3. 全く異なるデザインコンセプトを提案
4. ループの設定を変更（閾値、最大回数等）

どうしますか？
```

## スコア推移パターンと対応

### パターン1: 順調な上昇
```
15 → 18 → 21 → 24 → 26 (PASS)
```
REFINE を継続。5-6回で収束する理想的なパターン。

### パターン2: 早期PIVOT成功
```
15 → 16 → 16 (PIVOT) → 22 → 25 (PASS)
```
停滞を検出してPIVOT。新方向で急上昇。

### パターン3: 高原状態
```
15 → 20 → 22 → 22 → 22 (ESCALATE)
```
ある程度までは上がるが天井がある。ユーザーの方向性指示が必要。

### パターン4: 揺れる改善
```
15 → 20 → 18 (REVERT) → 21 → 24 → 26 (PASS)
```
一時的に悪化するが、REVERTで立て直し。最終的にPASS。

## PIVOTの実例（ブログ記事より）

### オランダ美術館サイト

**Iteration 1-9（REFINE）**:
- すっきりしたダークテーマのランディングページ
- 視覚的に洗練されているが、「予想通り」の範囲

**Iteration 10（PIVOT）**:
- 全く異なるアプローチ: サイトを**空間的体験**として再構築
- CSSパースペクティブで描画されたチェック柄の床を持つ3D空間
- 壁面に自由配置のアート作品
- ドアを介した部屋間ナビゲーション（スクロール/クリックではない）

**教訓**: PIVOTは「より良い同じもの」ではなく「根本的に異なるもの」を生む。
