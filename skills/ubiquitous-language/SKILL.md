# Ubiquitous Language - ユビキタス言語抽出

会話やコードベースからドメイン用語を抽出・整理し、曖昧さを検出して正規用語を提案する。

## プロセス

### 1. 用語のスキャン

以下のソースからドメイン関連の名詞・動詞・概念を収集する:

- **会話**: これまでの対話で使われた用語
- **コードベース**: モデル名、メソッド名、コメント（`explorer`サブエージェントで調査）
- **ドキュメント**: README、仕様書、PRD等

### 2. 問題の特定

以下の3種類の問題を検出する:

- **曖昧性**: 同じ単語が異なる概念に使われている（例: 「アカウント」が顧客と認証IDの両方を指す）
- **同義語**: 同じ概念に異なる単語が使われている（例: 「注文」「購入」「取引」）
- **曖昧な用語**: 定義が不明確で誤解を招く用語

### 3. 正規用語の提案

**意見を持って選ぶ**。複数の候補がある場合、最適な1つを選び、他は「避けるべきエイリアス」として明示する。

### 4. UBIQUITOUS_LANGUAGE.md の出力

作業ディレクトリに`UBIQUITOUS_LANGUAGE.md`を作成/更新する。

#### 出力フォーマット

```md
# ユビキタス言語

## [グループ名]（例: 注文ライフサイクル）

| 用語 | 定義 | 避けるべきエイリアス |
|------|------|---------------------|
| **注文** | 顧客が1つ以上の商品を購入するリクエスト | 購入、取引 |
| **請求書** | 配送後に顧客に送付される支払い要求 | 請求、支払い依頼 |

## [グループ名]（例: アクター）

| 用語 | 定義 | 避けるべきエイリアス |
|------|------|---------------------|
| **顧客** | 注文を行う個人または組織 | クライアント、購入者、アカウント |
| **ユーザー** | システム内の認証ID | ログイン、アカウント |

## 関係性

- **請求書**は正確に1つの**顧客**に属する
- **注文**は1つ以上の**請求書**を生成する

## 対話例

> **開発者:** 「**顧客**が**注文**すると、すぐに**請求書**を作る？」
> **ドメインエキスパート:** 「いや、**請求書**は**出荷**が確認されてから生成される。1つの**注文**から複数の**請求書**が出ることもある。」

## 検出された曖昧性

- 「アカウント」が**顧客**と**ユーザー**の両方の意味で使われていた — これらは異なる概念: **顧客**は注文を行い、**ユーザー**は認証IDである。
```

### 5. 複数コンテキストがある場合: CONTEXT-MAP.md の出力

**1つの語が複数の Bounded Context で異なる意味を持つ**場合、`CONTEXT-MAP.md` を別ファイルとして出力する。

#### 出力条件

以下のいずれかに該当する場合のみ作成:

- 同じ単語が**2つ以上のサブドメイン/コンテキスト**で異なる意味を持つ
- マイクロサービス境界をまたいで用語が翻訳されている
- 外部システム連携で用語の対応表が必要

単一コンテキストで完結する場合は作成不要。

#### 出力フォーマット

```md
# Context Map

> 各 Bounded Context での用語対応。同じ語が異なる文脈で異なる意味を持つことを明示する。

## Contexts

- **Sales Context**: 注文受付・見積もり
- **Fulfillment Context**: 在庫引当・出荷
- **Billing Context**: 請求・入金管理

## 用語対応表

| 用語 | Sales | Fulfillment | Billing |
|------|-------|-------------|---------|
| **顧客** | 見込み顧客含む | 配送先を持つ確定顧客 | 請求先（法人別管理） |
| **注文** | 引合 + 確定注文 | 引当済みの確定注文のみ | 請求書発行済みの注文 |
| **キャンセル** | 注文取消 | 出荷取消 | 返金処理 |

## 翻訳ルール

| From | To | 変換 |
|------|----|----|
| Sales.見込み顧客 | Fulfillment | （翻訳不可。Sales限定の概念） |
| Sales.確定注文 | Fulfillment.注文 | order_id を引き継ぎ |
| Fulfillment.出荷完了 | Billing.請求対象 | shipment_id → invoice_target |

## アンチコラプションレイヤー（ACL）

- **Sales→Fulfillment**: `OrderConfirmedEventHandler` で翻訳
- **Fulfillment→Billing**: `ShipmentCompletedEventHandler` で翻訳

## 検出された不整合

- 「キャンセル」が3コンテキストで意味が異なるのに、同名APIで露出されていた → 用語をコンテキスト名で修飾推奨（例: `cancelOrderInSales`, `cancelShipment`）
```

### 6. Flagged ambiguities（要確認の曖昧用語）

抽出中に**判断できなかった曖昧用語**を別セクションで明示し、ユーザー/ドメインエキスパートへの質問を添える。

#### 目的

- 「分かったつもり」で正規用語を決めない
- ドメインエキスパート不在時に勝手に決めず、保留状態を可視化
- 後続セッションで未解決事項を引き継ぎやすくする

#### 出力条件

以下のいずれかに該当する用語を Flagged ambiguities へ:

- 1つの単語に対して**2つ以上の解釈**が成立し、文脈から確定できない
- 既存コードと既存ドキュメントで**用法が矛盾**している
- 「業界標準では X だが、このPJでは Y の意味で使われている可能性」がある
- ステークホルダーごとに想定が異なる可能性が高い

#### 出力フォーマット

`UBIQUITOUS_LANGUAGE.md` の最後に追加:

```md
## Flagged ambiguities（要確認）

> 以下は解釈が確定できなかった用語。ドメインエキスパートに確認するまで暫定とする。

### 1. 「アカウント」

- **観測された用法**:
  - コードA: ログインID（auth context）
  - コードB: 顧客マスタの主キー（customer context）
  - 仕様書: 「アカウント解約 = サービス利用停止」
- **可能性のある解釈**:
  - (a) 認証ID と顧客は同一概念
  - (b) 別概念だが、解約フローでは同時に処理される
  - (c) 過去の名残で混在しており、整理が必要
- **暫定方針**: (b) と仮定して**ユーザー**（認証）と**顧客**（業務）に分離
- **確認したい質問**:
  - 1顧客に対して複数のユーザー（社員アカウント等）が存在しうるか？
  - 「アカウント解約」は片方だけ無効化するケースがあるか？

### 2. 「ステータス」

- **観測された用法**: 注文ステータス、配送ステータス、ユーザーステータス
- **可能性のある解釈**: 各ドメインで独立した state machine か、共通定義か
- **暫定方針**: ドメインごとに独立した型として扱う（Order.Status, Shipment.Status, User.Status）
- **確認したい質問**:
  - 「ステータス変更通知」のような横断機能で共通の型が必要か？
```

#### 利用方法

- ユーザーは Flagged ambiguities を見て、AskUserQuestionで返答
- 確定した内容は通常の用語表へ移動し、Flagged から削除
- 解決できない項目は次回起動時に再検討対象として残す

### 7. 出力前の確認

AskUserQuestionでファイル内容を提示し、承認を得てから書き込む（knowledge-management rulesに準拠）。

複数ファイル（`UBIQUITOUS_LANGUAGE.md` と `CONTEXT-MAP.md`）を出力する場合は、それぞれ個別に確認する。

## ルール

- **意見を持つ**: 同じ概念に複数の語がある場合、最適な1つを選び他はエイリアスとして列挙
- **衝突を明示**: 曖昧な用語は「検出された曖昧性」セクションで推奨を添えて明記
- **ドメイン用語のみ**: プログラミングの汎用概念（配列、関数、エンドポイント等）は除外。ドメイン固有の意味がある場合のみ含める
- **定義は簡潔に**: 1文以内。「何であるか」を定義し「何をするか」は書かない
- **関係性を示す**: 太字の用語名を使い、明らかな場合はカーディナリティも表現
- **自然なグループ分け**: サブドメイン、ライフサイクル、アクター等で自然にグループ化。無理にグループ分けしない
- **対話例を書く**: 3-5往復の短い対話で、用語の正しい使い方と概念間の境界を示す

## 再実行時の動作

同じ会話で再度呼ばれた場合:

1. 既存の`UBIQUITOUS_LANGUAGE.md`（と存在すれば`CONTEXT-MAP.md`）を読み込む
2. 新しい用語を取り込む
3. 理解が進化した場合は定義を更新（更新箇所に「(更新)」を付記）
4. 新しい曖昧性を再検出
5. **Flagged ambiguities** の各項目について、新情報で確定できるものは通常の用語表へ移動。残りは保留継続
6. 対話例を新しい用語を反映して書き直す
7. 新しいコンテキストが追加されたら CONTEXT-MAP.md の用語対応表と翻訳ルールを更新

完了後: 「この時点から、これらの用語を一貫して使用します。Flagged ambiguities に N 件の未解決事項があります。」と宣言する（未解決数を明示）。
