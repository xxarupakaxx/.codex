# Triage

プロジェクトの issue tracker にある issue を、小さな triage role の state machine に沿って進める。

この repo が外部 pull request を request の受付面として扱う場合（issue-tracker 設定を参照）、triage は PR も対象にする。
**PR は code が添付された issue である。**
同じ role、同じ state、同じ machine を使うが、違いは後述の「PR の場合」で示す。
tracker 設定に従って、`#42` のように種別が省略された参照を issue または PR として解決する。

triage 中に issue tracker へ投稿するすべての comment と issue は、次の免責文から**必ず**始める。

```
> *This was generated by AI during triage.*
```

## 参照文書

- [AGENT-BRIEF.md](AGENT-BRIEF.md)：長く使える agent brief の書き方
- [OUT-OF-SCOPE.md](OUT-OF-SCOPE.md)：`.out-of-scope/` knowledge base の仕組み

## Role

二つの **category** role：

- `bug`：何かが壊れている。
- `enhancement`：新機能または改善。

五つの **state** role：

- `needs-triage`：maintainer による評価が必要。
- `needs-info`：reporter からの追加情報を待っている。
- `ready-for-agent`：仕様が完全に定まり、AFK agent が着手できる。
- `ready-for-human`：人間による実装が必要。
- `wontfix`：対応しない。

PR では、同じ state を添付された code に対して読む。
`ready-for-agent` は brief が添付され、agent が diff に対する次の作業を行う状態を意味する。
`ready-for-human` は、人間が merge できる状態を意味する。

triage 済みの issue には、category role をちょうど一つ、state role をちょうど一つ付ける。
state role が競合している場合は、他の作業を始める前に問題を報告し、maintainer に確認する。

これらは標準 role 名であり、issue tracker で実際に使う label 文字列とは異なる場合がある。
mapping は事前に提供されている必要がある。
提供されていない場合は `/setting-up-engineering-skills` を実行する。

state transition は次のとおりである。
label のない issue は通常、最初に `needs-triage` へ進む。
そこから `needs-info`、`ready-for-agent`、`ready-for-human`、`wontfix` のいずれかへ進む。
`needs-info` は reporter が返信したら `needs-triage` に戻る。
maintainer はいつでも override できる。
通常と異なる transition に見える場合は、その点を報告し、先へ進む前に確認する。

## 呼び出し方

maintainer は `/triaging-issues` を呼び出し、望む作業を自然言語で説明する。
依頼を解釈して実行する。
例：

- 「確認が必要なものを見せて」
- 「#42 を確認しよう」（issue または PR）
- 「#42 を ready-for-agent に移して」
- 「agent が着手できるものは何？」

## 確認が必要なものを表示する

issue tracker を検索し、次の三つの bucket を古い順に示す。

1. **Unlabeled**：まだ triage されていない。
2. **`needs-triage`**：評価中。
3. **前回の triage note より後に reporter の活動がある `needs-info`**：再評価が必要。

PR が対象の場合は外部 PR も bucket に含め、各行へ `[PR]` または `[issue]` を付ける。
検索で表示するのは*外部* PR だけである。
外部の定義は tracker 設定に従う。
collaborator が作業中の PR は triage 対象ではない。
この filter は検索時だけに適用する。
明示的に指定された PR は author にかかわらず必ず triage する。

件数と、項目ごとの一行要約を示す。
maintainer に対象を選んでもらう。

## 特定の issue または PR を triage する

1. **コンテキストを集める。**
issue または PR の全体（body、comment、label、author、date、PR では diff も）を読む。
以前の triage note を解析し、解決済みの質問を繰り返さない。
プロジェクトの domain glossary を使い、対象領域の ADR を尊重して codebase を調べる。
codebase に対して二つの check を行う。
(a) **redundancy**：request の表現だけでなく domain concept でも検索し、要求された挙動の既存実装を探す。
探した場所を報告する。
見つかった場合は、実装済みの `wontfix` として step 5 へ進む。
(b) **prior rejection**：`.out-of-scope/*.md` を読み、この request と似たものがあれば提示する。

2. **推奨案を示す。**
推奨する category と state を理由とともに maintainer へ伝える。
すでに実装されているかどうかを含め、request に関係する codebase の短い要約も示す。
指示を待つ。

3. **主張を検証する。**
問いを重ねる前に、主張が正しいか確認する。
bug では reporter の手順から再現する。
PR では、checkout して関連する test または command を実行し、diff が主張どおりに動くか確認する。
結果を「確認済み（code path 付き）」「失敗」「詳細不足」のいずれかとして報告する。
詳細不足は `needs-info` の強い根拠になる。
検証済みの結果があれば、agent brief の確度が大きく上がる。

4. **問いを重ねる（必要な場合）。**
request の具体化が必要なら、`/grilling` と `/modeling-domains` skill を一緒に実行する。
一度に一問ずつ問い、domain 用語を明確にし、決定が確定するたびに `CONTEXT.md` と ADR をその場で更新して形を整える。

5. **結果を適用する：**

   - `ready-for-agent`：agent brief comment を投稿する（[AGENT-BRIEF.md](AGENT-BRIEF.md)）。
   - `ready-for-human`：agent brief と同じ構成を使うが、委任できない理由（判断、外部 access、design 上の決定、手動 test など）を記す。
   - `needs-info`：後述の template で triage note を投稿する。
   - `wontfix`：理由に応じた comment を付けて close する。
     - **実装済み**：変更は codebase にすでに存在する。
実装箇所を示す。
`.out-of-scope/` には書かない。
この knowledge base は*却下された* request 用であり、実装済みの request 用ではない。
     - **却下（bug）**：丁寧な説明を投稿して close する。
     - **却下（enhancement）**：`.out-of-scope/` へ書き、comment から link して close する（[OUT-OF-SCOPE.md](OUT-OF-SCOPE.md)）。
   - `needs-triage`：role を付ける。
途中まで進んでいる場合は、任意で comment を付ける。

## state をすぐに override する

maintainer が「#42 を ready-for-agent に移して」と指示した場合は、その判断を信頼して role を直接適用する。
実行する内容（role の変更、comment、close）を確認してから実行する。
問いは重ねない。
問いを重ねる session なしで `ready-for-agent` に移す場合は、agent brief を書くか確認する。

## Needs-info template

```markdown
## Triage Notes

**What we've established so far:**

- point 1
- point 2

**What we still need from you (@reporter):**

- question 1
- question 2
```

問いを重ねて解決した内容はすべて「established so far」に記録し、作業を失わないようにする。
質問は具体的で、回答によって行動できるものにする。
「追加情報をください」だけでは不十分である。

## 前の session を再開する

issue または PR に以前の triage note がある場合はそれを読む。
未回答だった質問へ reporter が答えたか確認し、続行前に更新後の状態を示す。
解決済みの質問を繰り返さない。
