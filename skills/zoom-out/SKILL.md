# Zoom Out - 鳥瞰モード

> 木を見て森を見ず、を防ぐための強制的な視点切替。

このスキルは **明示的な呼び出し時のみ** 起動する（自動推論で勝手に起動しない）。

## いつ呼び出すべきか

ユーザーが以下を感じたとき:
- 実装が長引いて全体像が見えなくなった
- 細かい議論で時間が溶けている
- 「これ本当にやる必要ある？」と疑問が湧いた
- 別アプローチがあるかも、と思った

## 出力フォーマット

呼び出されたら、以下5項目を **簡潔に** 出力する。長文化しない。

### 1. 元の目的（1文）

このタスクは **何を達成するため** に始めたか。

例: 「ユーザーが過去の注文をCSVでエクスポートできるようにする」

### 2. 現在地（3行以内）

今やっていること。

例:
- 進行中: CSV生成のフォーマッタ実装
- 残: API endpoint 接続、E2Eテスト
- 直近のブロッカー: 文字エンコード（UTF-8 BOM の有無）

### 3. 全体像（箇条書き）

タスク完了までに必要な全ステップを並べる:
- [ ] DBクエリ設計
- [x] CSV生成ロジック
- [ ] APIエンドポイント実装
- [ ] 認可チェック
- [ ] E2Eテスト
- [ ] ドキュメント更新

### 4. 代替案チェック（3つ）

「今のアプローチ以外に、他の選択肢はないか？」を3つ書く:

例:
- (a) 現状: バックエンドでCSV生成（実装中）
- (b) 別案: フロントでJSON→CSV変換（バックエンド変更不要）
- (c) 別案: 既存のレポート機能を流用（実装ゼロ、UI制約あり）

各案の短い trade-off を1行ずつ:
- (a) 大量データ対応OK / 実装多い
- (b) 実装少ない / 大量データで重い
- (c) 即完了 / カラム自由度なし

### 5. 次の意思決定（1つ）

ユーザーに確認すべき・決めるべき**1つの**意思決定を提示:

例: 「(a) を続けるか、(b) に切替えるか決めたい。エクスポート対象は最大1万行と聞いていたが、これは正しい？」

## ルール

- **5項目を超えない**: 鳥瞰なのに細かくしない
- **「現在地」は3行**: 詳細はメモリディレクトリに既にある
- **代替案は3つで打ち止め**: 5案出すと選びにくくなる
- **次の意思決定は1つ**: 複数あるなら優先度1位だけ提示

## アンチパターン

### 1. 詳細に逆戻りする Zoom Out

「現在地」のセクションでコード断片を貼らない。鳥瞰モードを破壊する。

### 2. 全代替案を実装する

代替案は **比較のため** に挙げる。実装コストが低い別案であっても、ユーザーが「切替える」と判断するまで実装しない。

### 3. 鳥瞰の連発

`zoom-out` を何度も呼ぶ＝意思決定が滞っている兆候。
2回目以降は AskUserQuestion で「決めるべきこと」を直接質問する。
