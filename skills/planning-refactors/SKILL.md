このスキルは、ユーザーがリファクタ依頼を作りたいときに起動します。
以下の手順に沿って進めます。
不要だと判断した手順は省略して構いません。

1. ユーザーに、解決したい問題と考えている解決案を、できるだけ長く詳しく説明してもらいます。
2. リポジトリを探索し、ユーザーの認識が正しいかを確かめ、コードベースの現在地を理解します。
3. 他の選択肢を検討したかを確認し、必要なら別案も提示します。
4. 実装について、漏れが残らないよう極めて詳細かつ徹底的にユーザーへ聞き取ります。
5. 今回変更するものと変更しないものを両方確認し、実装スコープの内外を正確に確定します。
6. その領域のテストカバレッジを調べます。不足している場合は、どのようなテストを追加する計画かをユーザーへ確認します。
7. 実装を極小コミットの計画に分解します。
8. その計画を GitHub Issue にまとめます。

手順 7 では、Martin Fowler の「各リファクタ手順は可能な限り小さくし、常にプログラムが動いていることを確認できるようにする」という助言を意識します。

Issue の説明には次のテンプレートを使います。

<refactor-plan-template>

## 問題の整理

開発者視点で見た、いま困っている問題を書く。

## 解決方針

開発者視点で見た、問題への解決方針を書く。

## コミット計画

長く詳細な実装計画を書く。
平易な言葉で書き、実装をできるだけ小さなコミット単位に分解する。
各コミットの終了時点で、コードベースは常に動作している状態を保つ。

## 意思決定メモ

実装にあたって決めたことを列挙する。
次のような内容を含めてよい。

- 作成または変更する module。
- 変更する module の interface。
- 開発者から得た技術的な補足。
- アーキテクチャ上の判断。
- スキーマ変更。
- API contract。
- 具体的な相互作用。

具体的なファイルパスやコード断片は入れてはいけない。
それらはすぐ古くなる可能性がある。

## テスト方針

決めたテスト方針を列挙する。
次の内容を含める。

- 良いテストの条件の説明。
- どの module をテストするか。
- コードベース内の先行事例。

良いテストとは、実装詳細ではなく外部から観測できる振る舞いだけを検証するテストである。

## スコープ外

今回のリファクタで扱わないことを書く。

## 補足メモ（任意）

その他の補足があれば書く。

</refactor-plan-template>
