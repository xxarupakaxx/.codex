# Diagnosing Bugs

> "Don't proceed to hypothesise without a loop."

## CRITICAL: フィードバックループ最優先

バグ修正で最も時間を浪費するアンチパターンは「ループを作らずに憶測で修正→確認に時間がかかる→憶測がズレていることに気づかない」。
**最初にやるべきは「観測の高速化」**。仮説検証1サイクルが30秒以内に収まる仕組みをまず作る。

## Phase 1: フィードバックループの構築（最優先）

### 1.1 ループの種類を選ぶ

`references/feedback-loop-recipes.md` に10種類のレシピを用意。バグの性質に応じて選択:

| バグの種類 | 推奨ループ |
|----------|----------|
| 単体ロジックの異常 | `failing test`（既存のテストランナーで赤く失敗するテストを書く） |
| HTTP API異常 | `curl/HTTP probe`（パラメータ変えながら反復） |
| CLI/スクリプト異常 | `CLI fixture`（小さい入力ファイル+1コマンド実行） |
| ブラウザ/UI異常 | `headless browser`（playwright-skillで自動化） |
| 本番でしか出ない | `replay trace`（ログ・トレースのリプレイ） |
| 既存コードに組み込めない | `throwaway harness`（最小再現プログラム） |
| 入力依存性が高い | `property/fuzz`（property-based testing） |
| いつ壊れたか不明 | `bisection`（git bisectで二分探索） |
| 既知の正解と比較したい | `differential`（既存実装と並行実行で差分検出） |
| 環境依存性が強い・自動化困難 | `HITL bash loop`（ユーザーが手で叩くテンプレ→AI観測） |

詳細: `Read references/feedback-loop-recipes.md`

### 1.2 HITL（Human-in-the-Loop）の判定

完全自動化が難しい場合:
- ループの実行はユーザーが手で行う
- 結果（出力・スクショ・ログ）をAIに渡す
- 1サイクルが30秒以内になるよう **コマンドは1行**, **出力はパイプで整形** を徹底
- テンプレ: `references/hitl-loop.template.sh`

### 1.3 ループ完成の判定基準（CRITICAL）

次のPhaseに進む前に確認:
- [ ] 1サイクルが30秒以内
- [ ] 「失敗」と「成功」が機械的に判定できる（exit code, テキストgrep, スクショ差分等）
- [ ] 環境依存（DBの状態、外部API）が固定化されている

満たせない場合は `Phase 1` から離れない。早すぎる仮説立ては禁止。

## Phase 2: 再現

### 2.1 最小再現の獲得

- ループを使い、**バグを必ず再現させる入力**を1つ確定
- 入力をfixtureとして固定化（再現の安定性）
- 「たまに再現する」場合はseed固定・並列度1・時刻固定でフレーキー要因を潰す

### 2.2 再現できない時

- ログ・スタックトレース・git logから**いつから・どの環境で**を絞る
- ユーザーに「再現環境のスナップショット」を要求（DB dump, 環境変数, バージョン）
- 推測で進めず、**Phase 1のループ設計に戻る**

## Phase 3: 二分探索で原因を絞り込む

### 3.1 探索軸

仮説を立てるより**機械的な絞り込み**を優先:
- **時間軸**: `git bisect`（いつから壊れた？）
- **コード軸**: 関連ファイルの片方を切り戻して再実行
- **データ軸**: 入力の半分を削って再実行
- **設定軸**: 環境変数を1つずつデフォルトに戻す

### 3.2 トレース注入

絞り込み中に追加観測が必要なら:
- `console.log`/`logger.debug`/`println` を該当箇所に注入
- **後で必ず削除**（CLAUDE.md禁止事項: 本番コードへのconsole.log残存）
- 一時的なトレースは `// TRACE-DIAGNOSE-XXXX` コメントで識別可能にしておく

## Phase 4: 仮説の検証と根本原因

### 4.1 仮説の3条件

仮説は次の3条件を全て満たす必要がある:
1. **観測されたすべての症状を説明できる**
2. **なぜ今まで動いていたかを説明できる**（回帰の場合）
3. **修正案が観測可能な変化をもたらすと予測できる**

### 4.2 検証

- 修正前にPhase 1のループで「修正後はどう変わるはず」を予測
- 修正適用→ループで予測通りか確認
- 予測と違うなら**仮説は間違い**。Phase 3に戻る

### 4.3 根本原因 vs 症状

- 表層の修正で済ませない
- 「なぜそのコードがそう書かれたか？」「他にも同じパターンがないか？」を確認
- 必要なら `compounding-knowledge` スキルで知見保存

## Phase 5: 修正と再発防止

### 5.1 テストで再発防止

- Phase 2で作った「最小再現」を**そのまま回帰テストに昇格**
- テストランナーで永続化（CLAUDE.mdワークフローのPhase 4品質確認に組み込む）

### 5.2 トレースの削除

- Phase 3.2で注入したトレースを全て削除
- `rg "TRACE-DIAGNOSE" --no-ignore` で漏れチェック

### 5.3 知見保存

非自明な原因だった場合は `compounding-knowledge` で保存:
- 症状の特徴（誰が引っかかりやすいか）
- 観測ループの設計（再利用可能なら）
- 根本原因と修正パターン

## アンチパターン

- **仮説ファースト**: ループなしで「たぶんここが怪しい」と修正→確認に5分→ハズレを繰り返す
- **過剰なトレース**: 全関数に `console.log` を撒く（観測点は二分探索の境界に集中）
- **症状の修正**: nullチェック追加で済ませて根本原因を放置
- **「直った気がする」で完了**: Phase 4.2の予測検証をスキップ
- **トレース消し忘れ**: コミット前の `rg "TRACE-DIAGNOSE"` を忘れる
