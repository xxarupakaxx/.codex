# Design an Interface

John Ousterhout「A Philosophy of Software Design」の「Design It Twice」原則に基づき、並列サブエージェントで根本的に異なるインターフェース設計を生成・比較する。

## brainstormingとの違い

| 観点 | design-an-interface | brainstorming |
|------|--------------------|---------------|
| 目的 | インターフェース形状の最適化 | ユーザー意図・要件の探索 |
| 手法 | 並列サブエージェントで複数案を同時生成 | 対話的に1案ずつ深掘り |
| 出力 | 型定義・メソッドシグネチャ・使用例 | 設計ドキュメント |
| 評価基準 | 深さ・単純さ・汎用性 | ユーザーのニーズとの合致 |

## プロセス

### 1. 要件収集

設計対象について以下を確認する:

- [ ] このモジュールが解決する問題は何か
- [ ] 呼び出し元は誰か（他モジュール、外部ユーザー、テスト）
- [ ] 主要な操作は何か
- [ ] 制約はあるか（パフォーマンス、互換性、既存パターン）
- [ ] 内部に隠すべきものと外部に公開すべきものは何か

不明点はAskUserQuestionで1つずつ確認。コードベースから判断できることは`explorer`サブエージェントで調査する。

### 2. 並列設計生成

Agentツールで**3つ以上のサブエージェント**を**同時に**起動する。各エージェントには**異なる制約**を課す:

```
各サブエージェントへのプロンプトテンプレート:

以下のモジュールのインターフェースを設計せよ: [モジュール説明]

要件: [収集した要件]

この設計の制約: [エージェントごとに異なる制約を割り当て]
- エージェント1: 「メソッド数を最小化 — 1-3メソッドを目指せ」
- エージェント2: 「柔軟性を最大化 — 多くのユースケースをサポートせよ」
- エージェント3: 「最も一般的なケースに最適化せよ」
- エージェント4: 「[特定のパラダイム/ライブラリ]からインスピレーションを得よ」

出力形式:
1. インターフェースシグネチャ（型/メソッド/パラメータ）
2. 使用例（呼び出し元がどう使うか）
3. この設計が内部に隠すもの
4. このアプローチのトレードオフ
```

**CRITICAL**: 各サブエージェントは根本的に異なるアプローチを取ること。似た設計は価値がない。

### 3. 設計の提示

各設計を順番に提示する:

1. **インターフェースシグネチャ** — 型、メソッド、パラメータ
2. **使用例** — 呼び出し元が実際にどう使うか
3. **隠蔽する複雑さ** — 内部に保持される実装詳細

ユーザーが各アプローチを消化できるよう、1案ずつ提示する。

### 4. 設計の比較

すべての設計を提示した後、以下の観点で比較する（表ではなく散文で）:

- **インターフェースの単純さ**: メソッドが少なく、パラメータがシンプル
- **汎用性 vs 特化**: 柔軟性とフォーカスのバランス
- **実装効率**: インターフェースの形がイディオマティックで効率的な内部実装を許容するか
- **深さ**: 小さなインターフェースが大きな複雑さを隠す = 深いモジュール（良い）。大きなインターフェースで薄い実装 = 浅いモジュール（避ける）
- **正しい使い方のしやすさ** vs **誤用のしやすさ**

### 5. 統合

最良の設計は複数のオプションの知見を組み合わせたものになることが多い:

- 「どの設計が主要なユースケースに最も合うか？」
- 「他の設計から取り入れるべき要素はあるか？」

ユーザーと合意した最終設計を明確に記述する。

## 評価基準（A Philosophy of Software Design より）

- **インターフェースの単純さ**: 少ないメソッド、シンプルなパラメータ = 学びやすく正しく使いやすい
- **汎用性**: 変更なしで将来のユースケースに対応できる。ただし過度な汎用化に注意
- **実装効率**: インターフェースの形が効率的な実装を可能にするか
- **深さ**: 深いモジュール = 良い設計の最重要指標

## アンチパターン

- サブエージェントに似た設計を生成させる（根本的な違いを強制する）
- 比較をスキップする（価値は対比にある）
- 実装に踏み込む（インターフェースの形状のみに集中する）
- 実装コストで評価する（インターフェースの質で評価する）
