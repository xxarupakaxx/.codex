# 完全無欠の問題解決（Bulletproof Problem Solving）7ステップ

ユーザーが持ち込んだ問題を、7ステップの方法論で**あなた（Codex）が分析する**ためのスキル。
ゴールは「それっぽい箇条書き」ではなく、本の規律に忠実な、意思決定に使える分析を出すこと。

## このスキルが普通の回答と違う点

汎用の回答は、問題を曖昧にしたまま思いついた打ち手を並べがち。このフレームワークの価値は、
そこを規律で防ぐところにある。常に次の5つを効かせる:

1. **問題を厳密に定義してから動く** — 何を解けば成功なのかを先に固定する。
2. **MECEなロジックツリーで分解する** — 漏れ・ダブりなく要素に割る。思いつきの列挙ではない。
3. **全部はやらない** — インパクトが大きく、かつ自分が動かせる枝だけに絞る（クリティカルパス）。
4. **ヒューリスティック（概算・経験則）から始める** — いきなり精緻な分析をしない。ざっくり桁を掴む。
5. **状況-複雑化-解決策のストーリーで締める** — 分析を、相手が動きたくなる筋書きに統合する。

このどれかを飛ばすと、ただの一般論に戻る。

## このスキルと既存スキルの関係

| スキル | 向き | 問い | 主な道具 |
|--------|------|------|---------|
| **bulletproof-problem-solving**（本スキル） | **前向き** | これからどう解く／何を選ぶ？ | 7ステップ・ロジックツリー・2×2 |
| `problem-analysis` | 後ろ向き | なぜ起きた？（原因・RCA） | A3 → Fishbone → 5 Whys |
| `brainstorming` | 拡散 | そもそも何を作る／意図は？ | アイデア探索・要件引き出し |

- **原因究明が主目的なら `problem-analysis`**、本スキルはその先（打ち手の選定・意思決定）を担う。
- 連結も有効: `problem-analysis` で根本原因を特定 → 本スキルで対策の優先順位とストーリーを組む。

このスキルは「考える道具」であり、コード変更ワークフローではない。会話中の分析タスクとして実行し、
CLAUDE.md の Phase 0-5（メモリディレクトリ作成・レビュー等）は**適用しない**。
分析が大きく資料化が必要なら、その時だけ Markdown/Word への落とし込みを提案する。

## 進め方の原則

- **7ステップは思考のエンジン、出力は読みやすい分析。** 下記テンプレに沿って各ステップの見出しは残しつつ、
  問題の大きさに応じて深さを伸縮させる。小さい問題ならステップ4〜6は数行に圧縮してよい。重い戦略課題なら厚く。
- **情報が足りなくても止まらない。** 意思決定に致命的に効く前提（誰の意思決定か／いつまでか／何をもって成功か／
  対象範囲）が欠けているときは、(a) 自分で妥当な前提を置いて「前提」として明記して進める。
  本当に答えが分岐してしまう1〜2点だけ、鋭い質問を返す。質問攻めにはしない（ユーザーは「分析してほしい」と言っている）。
- **概算には根拠を添える。** 「年間ざっと◯件 × 単価◯円 ≒ ◯円」のように、置いた数字と式を見せる。当てずっぽうにしない。
- **図が効くなら出す。** ステップ2のロジックツリーは、複雑なら Mermaid 図（mindmap か flowchart）で描くと伝わる。
  Mermaid Chart MCP（`mcp__claude_ai_Mermaid_Chart__validate_and_render_mermaid_diagram`）が使えるなら検証・描画に活用してよい。

> 一気通貫の記入例（定義文 → ツリー → 2×2 → 概算 → ストーリー）は `Read references/example.md` を参照。
> 出力に迷ったら必ず一度この実例を読んでから書くこと。

## ステップ1 ── 問題を定義する

最初に、関係者が合意できる「問題定義文」を1〜2文で作る。これが全ての土台。
覚え方は **SMART**: Specific（具体的）／ Measurable（測定可能）／ Actionable（実用的＝行動につながる）／ Relevant（関連性）／ Time-framed（時間枠）。

良い問題定義文の6つの特徴。満たすほど定義は強くなる（満たさない項目は前提で補う）:

1. **成果に焦点**。活動や中間成果ではなく、最終的な成果で書く。「会議を増やす」ではなく「解約率を◯%下げる」。
2. **具体的で測定可能**。何がどうなったら解けたと言えるかを、数値や状態で定義する。一般論にしない。
3. **明確な時間枠**。いつまでの話か。
4. **意思決定者の価値観・境界線に対処**。誰が何のために決めるか。求められる精度・望む規模・スコープの内/外を引く。
5. **創造性の余地を残す**。範囲を狭めすぎない。狭すぎる問いは解を人為的に縛り、良い打ち手を取りこぼす。
6. **できるだけ高いレベルで解く**。部分最適ではなく、組織・全体として解ける問いにする。

特徴4と5はぶつかりやすい——**境界は引くが、狭めすぎない**。このバランスの取り方が定義の腕の見せ所。

出力例:
> 問題定義文: 「(意思決定者) が (時間枠) 以内に、(対象範囲) において (成功条件＝測定可能な最終成果) を達成するには、どうすべきか。」
> 前提: （欠けていた点をここに明記）

## ステップ2 ── 問題を分解する（ロジックツリー）

問題を、分析できる構成要素／下位の問いに割る。思いつきの列挙ではなく **ロジックツリー** にする。

- 各枝は **MECE**（漏れなく・ダブりなく）を狙う。
- 「**切り口**（分割の仕方）」を意識する。同じ問題でも割り方で見える景色が変わる。
  価値が高そうなら **2つ以上の切り口** を試し、最も洞察が出る分け方を採用する。
  （例: 売上問題を「数量×単価」で割る切り口 と「新規×既存」で割る切り口、など。）
- 経済学・事業の枠組み（ユニットエコノミクス、需要と供給、コスト構造など）が、ドライバー特定の良いガイドになる。
- 必要に応じて、対立する仮説（"原因はAだ" vs "原因はBだ"）を枝として立てる。

出力: インデントしたツリー。複雑なら Mermaid の mindmap/flowchart で図示。

## ステップ3 ── 優先順位付けをする

全部の枝を分析しない。**2軸**で絞る:

- **縦軸: インパクトの大きさ**（その枝が問題解決にどれだけ効くか）
- **横軸: 自分が動かせる度合い**（その枝を自分・自社が実際に動かせるレバーを持つか）

両方が高い枝が **クリティカルパス**。ここに時間と人を集中させる。
出力: 簡単な2×2マップ（テキストでよい）か、優先度ランキング。「今回はこの2〜3点に絞る／残りは後回しにする理由」を明示する。

## ステップ4 ── 作業計画を立てる

優先した枝ごとに、答えを出すための計画を作る。Claude Code が実行できない部分（社内データ収集、現場確認など）は、
ユーザーが回せる計画として提示する。各項目に:

- **問い**（この枝で答えるべきこと）
- **必要なデータ／情報** と **入手方法**
- **良い答えの姿**（どんなアウトプットが出れば判断できるか）
- **おおまかな手間**（軽い／重い）

加えて、意思決定バイアスに注意を促す: **確証バイアス**（結論に都合の良い情報だけ集める）、
**サンクコストの誤謬**（過去の投資に引きずられる）、**アンカリングバイアス**（最初の数字に縛られる）。
多様な視点・専門家・反論役を入れる、上下関係をフラットにして本音を出す、といった運営の工夫も必要なら添える。

## ステップ5 ── 分析をする（ヒューリスティック先行）

**いきなり精緻な分析をしない。** まず単純なヒューリスティック（概算・経験則）で各要素の桁を掴み、
優先順位を素早く確かめる。多くの場合これで十分。

- 概算は式と前提を見せる（フェルミ推定の要領）。Claude Code が手元で計算できることは計算する。
- 重い手法（回帰分析、モンテカルロ、ゲーム理論、機械学習など）は **本当に必要なときだけ** 推奨し、
  「どの問いに・なぜその手法か」を一言添える。たいていは不要。
- 結果は「**状況・複雑化・解決策**」の形に要約しながら進め、仮説として検証していく。
- クリティカルパスから外れない。脇道の精緻化はしない。

## ステップ6 ── 分析結果を統合する

個々の分析が出ても、そこで終わりではない。バラバラの発見を **一つの論理構造に組み立て**、
「だから、こう解くべきだ」という推奨を支える形にする。

- 各分析結果が推奨を支えているか、筋が通っているかを点検する。
- 「結局なにが言えるのか（So what）」を一段上の主張として立て、その下に根拠を並べる。

## ステップ7 ── ストーリーで語る

問題定義文と、ステップ6で統合した結論をつないで **筋書き** にする。

- 全体を貫く考えとして **状況（Situation）→ 複雑化（Complication）→ 解決策（Resolution）** を使う。
  - 状況: 誰もが同意する前提・背景
  - 複雑化: それを揺るがす問題・変化・緊張
  - 解決策: だからこうする、という推奨
- 統合した分析を、帰納または演繹の論理で支持する根拠として並べる。
- **行動につなげて締める**: 推奨アクションと次の一歩。意思決定者の受容度に応じて、断定するか、
  動機づける問いを投げる形にする。

## 出力テンプレート

問題を分析するときは、原則この構成で返す（深さは問題規模に応じて伸縮）:

```
## 問題定義
（1〜2文の問題定義文。欠けていた前提があれば「前提:」で明記）

## 分解（ロジックツリー）
（MECEなツリー。複雑なら図。採用した切り口とその理由）

## 優先順位（クリティカルパス）
（インパクト×可動性で絞った2〜3点。後回しにするものとその理由）

## 作業計画
（優先した枝ごとに、問い／必要情報／良い答えの姿。Claude Codeが概算できる所は次で実施）

## 分析（ヒューリスティック先行）
（概算と前提・式。必要な所だけ深掘り。状況・複雑化・解決策で要約）

## 統合
（発見をつないだ「だからこう解く」という主張）

## ストーリーと打ち手
（状況→複雑化→解決策の筋書き＋推奨アクション＋次の一歩）
```

最後に必要なら、「もっと深掘りしたい枝」「資料化（Word等）」「ロジックツリーの図示」を一言で提案する。
ただし提案は短く、本筋を埋めない。

## 使わない場面

- ❌ 単純な事実質問・調べ物（即答できるもの） → 普通に回答する
- ❌ すでに起きた事象の原因究明・RCA（「なぜ落ちた？」） → `problem-analysis`
- ❌ 純粋なアイデア出し・要件/意図の探索 → `brainstorming`
- ❌ バグの再現・コールスタック追跡 → `diagnosing-bugs` / `root-cause-tracing`
- ❌ コードのアーキテクチャ改善 → `improving-architecture`

判断に迷う問題（前向きの意思決定か、後ろ向きの原因分析か曖昧）なら、まず本スキルの
ステップ1で問題定義文を書く——「成果（前向き）」で書けるなら本スキル、「原因（後ろ向き）」が
主眼なら `problem-analysis` に切り替える。
