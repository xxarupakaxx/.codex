# Grill Me

計画や設計について、決定木の各ブランチを1つずつ歩き、判断の間の依存関係を1つずつ解決していく徹底的なインタビューを行う。

ゴールは**共通理解**の形成。すべてのブランチが解決されるまで質問を続ける。

## ルール

- **1回に1問だけ**質問する（まとめて聞かない）
- 各質問に**推奨回答を添える**（「私ならこうする。理由は〜」）
- **コードベース調査で答えられる質問は聞かずに自分で調べる**（`explorer`サブエージェントを使用）
- 依存関係のある質問は、依存先を先に解決する
- ユーザーの回答に矛盾があれば即座に指摘する

## Optional Phase: CONTEXT / ADR 連動（重要判断が含まれる場合）

質問を進める中で**設計の前提**や**重要判断**が確定したら、その都度ドキュメントへ反映する:

### 1. 前提・制約が判明した時 → CONTEXT.md

`docs/CONTEXT.md` または `${MEMORY_DIR}/memory/<task>/context.md` に追記。
記録対象:
- システムの前提条件（「DBは PostgreSQL 15+」等）
- 業務上の制約（「ユーザーは1日10回まで」等）
- 既存システムとの境界（「認証は外部 IdP に委譲」等）

### 2. 用語の曖昧さが判明した時 → ubiquitous-language

同じ概念に複数の名前、または1つの名前で複数の概念を指している場合は
`ubiquitous-language` スキルを呼び出し、用語集に登録する。

### 3. 重要判断が確定した時 → ADR

以下の3条件を**全て**満たす判断のみADR化（`~/.claude/rules/adr-criteria.md`参照）:
1. **Hard to reverse**: 後戻りが困難（DB schema, API 仕様, 言語選定など）
2. **Surprising without context**: 背景なしには驚かれる選択
3. **Result of real trade-off**: 複数案を実質比較した結果

該当するなら `creating-adr` スキルでADR化を依頼。

### 4. タイミング

- 質問1問ごとに即時記録ではなく、**1つの決定木ブランチが解決した時点**で反映
- ユーザーに「今の合意をCONTEXT/ADRに記録していい？」と確認してから書く（CLAUDE.md知見ファイルEdit禁止ポリシーに準拠）
